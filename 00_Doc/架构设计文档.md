> 基线参照：原始可用版本（`../ZDT_Motor_system 原始可用版本`）、架构优化版本（已起草 `handler/chassis_service`）。

## 0. 目标与范围

### 0.1 本次要解决的问题

当前「原始可用版本」功能完整，但架构上有四个痛点：

1. **`freertos.c` 是"上帝文件"**（≈680 行）：RTOS 任务定义、命令队列、导航仲裁、里程计 glue、地图上报、编码器增量计算、BLE 轮询全部混在一起。
2. **缺少显式"适配层"**：跨层是裸调用，参数校验零散，错误码无法逐层上抛。
3. **抽象不一致**：`Bsp_ZDT_UART` 用函数指针注入（标准做法），但 `Bsp_HWT101` 直接 `#include "stm32f7xx_hal.h"`，把平台类型焊进了 BSP 头文件；`Pid_port` 也 `#include "main.h"`。
4. **命名/目录不统一**：`Chssis`（拼写错）、`ZDT-UART`（连字符）、`PID_ControllerTypeDef`（驼峰）与 `bsp_zdt_*`（下划线）并存。

### 0.2 本次交付

**仅交付本设计文档**，不改动代码。文档对齐后，再按第 9 章的迁移计划分阶段实施。
每个模块的落地顺序固定为：**先共同设计 `.h`（接口 + 命名定稿）→ 再写 `.c`**。

### 0.3 已确认的决策（评审后）

| 决策点 | 结论 |
|---|---|
| 适配层形态 | **每模块拆双文件**：纯逻辑 `xxx.c` + 适配 `xxx_adaption.c/h` |
| 适配层覆盖面 | **一个不漏**，每个项目模块都有适配层（含 map/pid） |
| 本轮产出 | **只要架构设计文档**，不动代码 |
| 目录与命名 | **按层重组 + 统一下划线小写**；名字**缩短但保留原特征**，最终名写 `.h` 时定稿 |
| 层编号 | `01_App … 04_Bsp` 自顶向下 |
| OS 层 | **本次不剥离**，Service 直接用 CMSIS-OS2/FreeRTOS；OS 抽象列为未来可选 |
| 中断处理 | `it_dispatch` 作为**中断适配层**，放 **Service 层**（评审改） |

## 1. 总体分层架构

沿用立芯教学框架的分层骨架（APP / 中间件 / OS / BSP / Core / Driver），并为本项目补上一个**服务/编排层**（解决 `freertos.c` 上帝文件问题）。每相邻两层之间的"那根带标注的箭头"，在代码里实体化为**适配层**。

```
┌──────────────────────────────────────────────────────────────┐
│  01_App        业务流程 / 模式状态机 / 上电时序                │  ← 决定"做什么"
│                app_main、模式切换、上报格式                    │
└───────────────┬──────────────────────────────────────────────┘
                │  ▲ 命令(post_xxx) / 状态查询          [适配=参数校验+语义封装]
┌───────────────▼──────────────────────────────────────────────┐
│  02_Service    任务 / 命令队列 / 导航仲裁 / 组装根             │  ← 决定"何时、并发"
│                chassis_service、system_assembly               │
│                it_dispatch ← 中断适配层(独占 HAL ISR 回调)     │
│                (注入 OS 接口给 BSP handler；直接用 CMSIS-OS2)  │
└───────────────┬──────────────────────────────────────────────┘
                │  ▲ 算法调用(运动学/寻路/PID)      [适配=参数校验+场景参数+单位转换]
┌───────────────▼──────────────────────────────────────────────┐
│  03_Middleware 纯算法：运动学 / 里程计 / A*寻路 / PID         │  ← 平台无关，参数化
│                mecanum_chassis、map、pid (各带 _adaption)      │
└───────────────┬──────────────────────────────────────────────┘
                │  ▲ 设备操作(收发/控速/读角)          [适配=参数校验+接口注入]
┌───────────────▼──────────────────────────────────────────────┐
│  04_Bsp        板级外设对象：电机 / 陀螺仪 / 蓝牙             │  ← 对象化驱动
│                zdt_motor(+zdt_cmd+handler)、hwt101、ble        │
│               (handler 独占总线串行 I/O；_adaption 暴露 rx_isr)│
└───────────────┬──────────────────────────────────────────────┘
                │  ▲ IO控制(UART/DMA/GPIO/Tick) + ISR  [适配=port函数+句柄绑定]
┌───────────────▼──────────────────────────────────────────────┐
│  Core          STM32 HAL + 外设 init + 时钟 + MPU             │  ← CubeMX 生成，保留
│                FreeRTOS 内核也在此层之下被 OS 直接调用         │
└───────────────┬──────────────────────────────────────────────┘
                │
┌───────────────▼──────────────────────────────────────────────┐
│  Drivers       CMSIS + STM32F7xx_HAL_Driver（ARM/ST，保留）   │
└──────────────────────────────────────────────────────────────┘
```

**铁律：依赖只能从上往下，下层永不 `#include` 上层。** 跨层调用必须经过被调方的**适配层 API**，禁止上层直接调用下层的纯逻辑函数。

> **关于 OS**：本次不单独抽 OS 适配层。Service 层与中断回调直接使用 CMSIS-OS2（`osMessageQueue*`/`osMutex*`/`osThread*`）和必要的 FreeRTOS `FromISR` 接口。等后续需要换 RTOS 或上裸机时，再把这些调用集中抽成 `os_port`（见第 11 章未来项）。

## 2. 核心约定：适配层（双文件）模式

这是全项目最重要的约定。**每个项目自研模块拆成两个文件对**：

| 文件 | 角色 | 能 `#include` 什么 | 不能做什么 |
|---|---|---|---|
| `xxx.c / xxx.h` | **纯逻辑核心** | 仅标准库（`stdint/stdbool/stddef/string/math`）与本模块头/配置头 | ❌ 不碰 HAL、不碰 FreeRTOS、❌ 不做 IO |
| `xxx_adaption.c / xxx_adaption.h` | **适配层（层边界）** | HAL、OS、下层模块、本模块纯逻辑头 | ❌ 不写算法/协议细节 |

### 2.1 适配层适配的是三类东西（重要）

"适配层"不只适配平台。不同层的适配层，适配对象不同——但职责形态一致（校验 + 转换 + 注入）：

| 适配类型 | 所在层 | 适配什么 | 例子 |
|---|---|---|---|
| **平台适配** | Bsp | 注入 HAL/IO 能力（UART/DMA/GPIO/Tick）、ISR | `zdt_motor_adaption`、`hwt101_adaption`、`ble_adaption` |
| **场景/参数适配** | Middleware | 装入"本机 / 本场地"的具体参数与标定 | `map_adaption`（场地尺寸/分辨率/障碍）、`pid_adaption`（增益）、`chassis_adaption`（轮径轴距） |
| **并发/装配适配** | Service | 队列、任务、互斥、对象组装与注入 | `system_assembly`、`chassis_service` |

> 这正是第 3 个评审点的结论：**map 的精度（分辨率、膨胀半径、场地尺寸、障碍布局）和实际场景强相关**，所以它必须有适配层来承载场景参数，纯算法核心 `map.c` 只保留"通用栅格 + A* 引擎"。**没有"纯算法就不要适配层"的例外。**

### 2.2 纯逻辑核心（`xxx.c`）职责

- 只实现**算法 / 协议 / 状态机**本身，**完全平台无关、可单元测试、可移植**。
- 所有平台能力通过**对象内的函数指针**获得（依赖注入），自己不知道底层是 UART 还是 SPI、是 FreeRTOS 还是裸机。
- 所有场景参数通过**入参 / 配置头**获得，自己不内嵌"本场地"的魔数。
- 只做**内部不变量**检查（可用 `assert`）；对外 API 可做轻量防御性 NULL 检查，但**不重复**适配层已做的范围校验。

### 2.3 适配层（`xxx_adaption.c`）职责 —— 四件事

1. **参数校验**（用户最关心的点）：本层是该模块对上的**唯一合法入口**，对所有外部输入（上层入参、ISR 数据、协议报文）做显式校验，**不合法立即 `return` 状态码**。
2. **依赖 / 接口注入**：实现下层/平台的 port 函数（如 HAL UART 收发），在 `xxx_adaption_init()` 里调用纯逻辑的 `xxx_inst(...)` 把函数指针装进对象。
3. **场景参数适配**：把"本机/本场地"的配置（尺寸、增益、标定、障碍布局）喂给纯逻辑核心。
4. **单位 / 坐标转换**：deg↔rad、pulse↔mm、rps↔电机原始速度、协议坐标↔里程计坐标。

> **参数校验放哪？** 跨信任边界的校验**集中在适配层**；纯逻辑层只管内部不变量。这样既满足"每层适配层做参数检查"，又不出现两层重复校验。

### 2.4 参数校验写法（强制）

```c
/* 适配层 API 入口：if + return，不封装成独立函数 */
chassis_status_t chassis_set_velocity(float vx, float vy, float wz)
{
    if (g_inited == 0U) {                    /* 标志位检查 */
        return CHASSIS_ERR_NOT_INIT;
    }
    if (vx > VX_MAX || vx < -VX_MAX) {       /* 范围检查 */
        return CHASSIS_ERR_PARAM;
    }
    /* ……转换 + 转调纯逻辑…… */
}
```

- NULL / 长度 / 范围 / 标志位等**简单检查不要封装成函数**（除非逻辑 ≥5 行或明显复用）。
- 不用 `assert` 校验外部输入；`assert` 只校验内部不变量。
- 返回值统一用本模块的 `xxx_status_t` 枚举（见第 7 章）。

### 2.5 依赖注入 / 跨层接口契约

把图中"层间箭头"写成**接口结构体（函数指针）+ 构造函数 `_inst` + 组装根**：

```c
/* 纯逻辑头 hwt101.h：声明"我需要什么"，但不关心由谁实现 */
typedef struct {
    hwt101_status_t (*pf_start_rx)(void *ctx);   /* 启动接收 */
    hwt101_status_t (*pf_stop_rx)(void *ctx);    /* 停止接收 */
} hwt101_io_if_t;

typedef struct {
    hwt101_status_t (*pf_get_ms)(uint32_t *out); /* 时基 */
} hwt101_time_if_t;

hwt101_status_t hwt101_inst(bsp_hwt101_t *self,
                            const hwt101_io_if_t   *io,
                            const hwt101_time_if_t *time,
                            void *io_ctx);
```

- **谁需要 → 在自己头文件里定义接口结构体。**
- **谁实现 → 下层/适配层填函数指针。**
- **谁组装 → 组装根 `system_assembly.c`（唯一允许见到所有具体类型的地方）调用各 `_inst`。**

这套写法 `Bsp_ZDT_UART` 已经在用（`pf_send_cmd`/`pf_start_receive`），本次把它**统一推广到每个模块**。

### 2.6 示例 A：HWT101 拆分（平台适配）

HWT101 是"平台耦合焊进 BSP"的典型。

**拆分前（现状）** —— 纯逻辑头里直接出现 HAL 类型：

```c
/* Bsp_HWT101.h（现状）：BSP 头被 HAL 污染 */
#include "stm32f7xx_hal.h"
typedef struct {
    UART_HandleTypeDef *p_uart;     /* ❌ HAL 类型泄漏进对象 */
    DMA_HandleTypeDef  *p_dma_rx;
} hwt101_port_t;
```

**拆分后** ——

```text
04_Bsp/hwt101/
├── hwt101.c / hwt101.h                  纯解包状态机；零 HAL；port 用 void*ctx + 函数指针
└── hwt101_adaption.c / hwt101_adaption.h  含 HAL；实现 pf_start_rx/pf_stop_rx/pf_get_ms；
                                            参数校验；调 hwt101_inst() 注入
```

好处：`hwt101.c` 不再 `#include "stm32f7xx_hal.h"`，可在 PC 上单测解包逻辑；换芯片只改 `hwt101_adaption.c`。ZDT 电机已是这个结构，只需把 `System_adaption.c` 里的 `zdt_motor*_send/receive`（HAL UART DMA 包装）下沉到 `zdt_motor_adaption.c`。

### 2.7 示例 B：map 拆分（场景/参数适配）

map 的"精度与场景相关"决定了它的拆法：

```text
03_Middleware/map/
├── map.c / map.h                  通用栅格 + A* 引擎；静态数组按 map_cfg.h 编译期常量定尺寸；
│                                   通用 map_set_obstacle_* / map_find_path
└── map_adaption.c / .h (含 map_cfg.h)
       ├─ 场景常量：分辨率、场地尺寸、障碍膨胀半径(=车半径)、软代价半径
       ├─ map_load_field_obstacles()  ← 本届赛场的具体障碍布局（原 map_load_default_obstacles）
       └─ mm 坐标原点偏移            ← 原 freertos.c 的 CHASSIS_PATH_TARGET_OFFSET_X/Y
```

- 纯 `map.c` 不再硬编码"本赛场"的障碍和偏移；换场地只改 `map_adaption.c` 与 `map_cfg.h`。
- 注意：静态数组尺寸需编译期常量，所以分辨率/场地尺寸放 `map_cfg.h`（被 `map.c` include），不是运行期注入。

### 2.8 中断适配层：`it_dispatch`（放 Service 层）

中断是从 Core（HAL）冒上来的横切事件，需要被适配/路由到对应模块。`it_dispatch` 就是这层适配：

- **独占** HAL 的 `HAL_UARTEx_RxEventCallback` / `HAL_UART_ErrorCallback` 等回调（全工程唯一定义点）。
- 只做**轻量适配**：按 `huart` 句柄判断来源 + 长度边界检查，然后调对应模块的 ISR 入口（如 `hwt101_on_rx`、`zdt_motor_on_rx`、`ble_on_rx`）。
- **重活（解析/入队/重启接收）留在各模块自己的 `_adaption`**，dispatch 保持瘦。
- ISR 上下文**禁止阻塞**；与任务共享状态走 `FromISR` 队列 / 互斥（沿用现状并审查）。

**为什么放 Service 而不是 Bsp**（评审结论）：
1. 它持有的"哪个 `huart` 对应哪个模块"是**系统接线知识**，和 `system_assembly` 同源——接线信息集中在一层更内聚。
2. 它的回调要往 **Service 自己的队列**（如电机上报队列）`FromISR` 投递，这是并发职责，本就属 Service。
3. 这样 **Bsp 模块保持纯粹**：只暴露 `xxx_on_rx` 入口，不需要知道自己挂在哪个 UART 上。Service→Bsp 是向下调用，合法。

### 2.9 handler 模式：总线独占 + 仲裁（电机子系统）

适配层解决"单次调用"的跨层校验与注入，但**当多个对象共享一条总线、且对外有并发访问**时，还需要一层 **handler** 来做运行时仲裁。这是 EternalChip `bsp_led_handler` 的同款模式，本项目用在电机子系统（`zdt_motor_handler`）。

**为什么电机需要 handler（而 hwt101/ble 不需要）**：

| 约束 | 来源 | 结论 |
|---|---|---|
| 4 台电机 TX **共用一条总线**（原 huart3） | 硬件接线 | 必须串行化，单一发送者 |
| 每帧之间电机要求 **≥3ms 处理间隔** | 电机固件 | 发送需 tick 门控节流，非"等 DMA 完成" |
| 上层多处会下发命令（控制环、模式机） | 软件并发 | 命令要排队仲裁，不能并发直发 |
| 位置 `int64` 被 RX 写、里程计读 | 跨线程 | 需临界区保护，避免撕裂 |

> hwt101/ble 各自独占一条 UART、无共享总线、无对外并发下发，所以适配层足矣，不需要 handler。

**handler 三件事**：

1. **总线独占（防黏包）**：handler 拥有的是**一条总线**而非"一个组"——同总线全部电机（4 轮 + 将来夹爪）都归它管。它是该总线唯一发送者，发送间用 `last_tx_tick` 门控强制 ≥`gap_ms`（3ms）。
2. **TX/RX 双线程**：
   - **TX 线程**：从 `tx_queue` 取上层投递的命令 → 补够 3ms 间隔 → 串行下发。
   - **RX 线程**：从 `rx_queue` 取中断喂入的原始帧 → `zdt_motor_parse` 解析 → 临界区写 `pos_pulse`。
   - 拆两线程的原因：上报 2ms/帧 × 4 ≈ 2000 帧/s 的 RX 解析负载，不能被 TX 的 3ms 节流拖延。
3. **OS 接口注入（保持可移植）**：handler 本身 OS 无关，线程/队列/时间/临界区经**分离的接口结构体**（`mh_os_thread_t`/`mh_os_queue_t`/`mh_os_time_t`/`mh_os_lock_t`）注入；具体 CMSIS-OS2 实现由 `system_assembly` 提供。这是"本次不剥离 OS 层"前提下，仍让 BSP handler 可移植的折中——OS 调用集中在装配层那几个薄包装里。

**单机是一等公民**：`mh_enable / mh_pos / mh_home / mh_get_pos` 对总线表里任意 `idx` 都能单独操作；`mh_speed(speed, n, accel)` 只对前 n 个（=4 轮）做同步广播，夹爪（idx≥4）用 `mh_pos` 单独控，不卷进速度帧。

**数据流**（详见 `04_Bsp/zdt_motor/README.md`）：
```
上层 → mh_speed/mh_pos/...  → tx_queue → [TX线程] 3ms门控 → HAL UART DMA
中断 → it_dispatch → zdt_adp_rx_isr(拷贝帧) → mh_feed_rx → rx_queue
                                          → [RX线程] zdt_motor_parse → pos_pulse
里程计任务 → mh_get_pos(临界区读 pos_pulse)
```

> handler 放在 `04_Bsp/zdt_motor/`（而非 Service）：它是电机子系统的一部分，与驱动一起整体复用；只有“OS 具体实现”留在 Service 注入。这与 §2.8 的 `it_dispatch` 放 Service 不矛盾——dispatch 是全局接线知识，handler 是单设备子系统的内部并发。

### 2.10 上电时序握手：RX 事件钩子 + Service 编排

handler 解决了"总线并发与节流"，但还有一个**上电时序**问题：实际车上**板子与电机上电先后不可控**（队友常板子先上电）。若 `system_assembly` 在初始化时无条件、立即下发"使能 + 设置上报"，而此刻电机尚未上电，这些命令会**直接丢失**；电机随后上电时身上没有上报配置，一旦上层下发速度命令，电机就在**没有闭环反馈**下运动，有撞坏风险。

**设计原则（延续 handler 无状态）**：

| 角色 | 职责 | 不做什么 |
|---|---|---|
| **handler（Bsp）** | 纯执行器 + **事件源**：加一个 RX 事件钩子 `mh_on_event(pf, ctx)`，RX 线程每解析出一帧（`zdt_motor_parse` 的 `rx.kind/code/pulse`）**出锁后**经 `pf_on_evt(ctx, idx, &rx)` 上抛 | ❌ 不存"就绪状态"；❌ 不做握手编排；❌ 运动接口不加"未就绪拦截" |
| **Service（system_assembly）** | **就绪状态 + 握手编排**：钩子置位本层全局 `g_enabled`(收 ACK) / `g_reporting`(收位置上报)；`init` 起线程后进入编排循环——对未就绪电机重发使能 + 上报，**直到全部电机回上报帧才放行** | ❌ 不碰 handler 内部 |

> **为什么就绪状态不放 handler**：就绪是"系统编排状态"，属 Service；handler 只管执行与上抛，保持 OS 无关、可移植、无状态——与 §2.9 的注入式设计一致。「使能」与「上报」在 Service 分开记录，放行只认 `g_reporting`（收到位置上报帧 = 闭环真在跑的硬证据），`g_enabled`(ACK) 仅作诊断。

**握手数据流**（补充 §2.9）：

```
[上电] system_assembly_init（须在调度器启动后的线程上下文）:
   mh_inst → mh_on_event(motor_evt_cb) → mh_start → zdt_adp_start
   → for(;;){ 对未就绪电机 mh_enable + 整组 mh_report;
              g_reporting 全置位 ⇒ break 放行; 否则 osDelay 后重试 }
[RX线程] zdt_motor_parse → 出锁 → pf_on_evt(idx,&rx)
   → motor_evt_cb: rx.kind==ZDT_RX_POS ⇒ g_reporting[idx]=1（首帧打一条 LOGI）
[放行] 全部 g_reporting=1 ⇒ g_assembled=1（此前 system_motor_handler() 返回 NULL）
```

**关键特性与约束**：

- **与上电顺序无关**：电机晚于板子上电，也会在其上电后被编排循环纳管（命令一直重发，幂等）。
- **杜绝裸跑**：闭环建立前不放行 `g_assembled`，业务层经 `system_motor_handler()` 拿不到 handler，无法下发运动——这是"防止无反馈乱跑损坏车"的根本保障。
- **调用上下文约束（重要）**：`system_assembly_init` 现在会**阻塞**到全部电机就绪，**必须在调度器启动后的线程上下文调用**（依赖 `osDelay` 让出 + RX 线程运行）。当前由 `freertos.c` 的 `Main_task` 调用，满足；将来切到 `app_main.c::app_init` 入口时须删除 `freertos.c` 内的重复调用，避免 `mh_start` 二次起线程。
- **默认无限等（安全优先）**：某电机始终不上线则系统不放行，RTT 周期打印 `waiting handshake: N motors`，便于定位卡在哪几台。

**已知局限（后续）**：`g_reporting` 一旦置位不回退——电机运行中掉电仍判为就绪。后续可加 `last_rx_tick` + 上报超时降级（运行期看门狗）。hwt101/ble 的同类上电握手可复用本模式。

## 6. 统一编码规范要点

完整规范见仓库内 `嵌入式代码规范标准.md` 与立芯 skill。下面只列**本次重构强制执行**且**对旧文档有更新**的条目（冲突时 Codex 规则优先）。

### 6.1 相对旧 `嵌入式代码规范标准.md` 的更新点 ⚠️

| 项 | 旧文档 | **本次采用（Codex 优先）** |
|---|---|---|
| 命名法 | 匈牙利命名法（下划线隔开） | **禁用匈牙利命名**，统一简洁 `snake_case` |
| 行宽 | ≤80 列 | **≤80 列（汉字按 2 列计）** |
| 文件头 | 含 author/version/date | **≤10 行 Doxygen，保留 @author，去 version/date** |
| 变量名长度 | 未限 | **≤15 字符**（优先 8–12） |
| 函数名长度 | 未限 | **≤20 字符**（优先 12–18） |
| 宏名长度 | 未限 | **≤25 字符** |

> 注：保留 `@author` 是本项目对 Codex「文件头禁作者」的有意偏离，便于驱动署名与复用；version/date 仍交给 git，不写进文件头。

### 6.2 不变的硬性要求

- 缩进 4 空格，禁 TAB；K&R 大括号；`if/for/while/switch` 一律带大括号。
- **禁止动态内存**（`malloc/free`）、**禁止 VLA**。
- 任何可能失败的调用（HAL/OS/项目接口）**调用后立即检查返回值**。
- **超时基于 SysTick / 统一 ms 计数**，禁空循环计数。
- 外设名通过宏/配置表表达，业务里**不硬编码 `USART1`**。
- 局部变量声明处带行尾用途注释；**注释密度 >30%**；单函数内部注释 ≤5 条。
- 能 `static` 就 `static`，尽量减少全局；中断/多线程共享状态必须写明同步策略。
- 函数尽量 ≤50 行、嵌套 ≤4 层，一个函数只做一件事。

### 6.3 命名：缩短 + 保留特征（写 `.h` 时定稿）

原则：**名字缩短、不堆长单词，但保留能认出模块/动作的特征。** 下表是方向示意，**不是最终名**——每个模块的最终名在我们一起写 `.h` 时一并敲定：

| 现有名 | 长度 | 方向（示意，待定稿） |
|---|---|---|
| `bsp_zdt_motor_group_set_periodic_position_report` | 48 | `zdt_grp_set_report` |
| `bsp_zdt_motor_set_speed_profile` | 31 | `zdt_set_speed` |
| `mecanum_chassis_set_velocity_radius` | 35 | `chassis_set_radius` |

> 策略：前缀用缩写（`zdt_`/`hwt_`/`chassis_`/`map_`），动词用短名（`set/get/run/init`）。模块前缀会吃掉长度预算，定稿时按"≤20 字符且能认出"权衡。

## 7. 错误码 / 状态枚举统一约定

各模块已各有 `xxx_status_t`，本次**统一成员语义与顺序**（值可不同，但语义一致，便于适配层逐层映射）：

```c
typedef enum {
    XXX_OK        = 0,   /* 成功 */
    XXX_ERR       = 1,   /* 通用错误 */
    XXX_ERR_PARAM = 2,   /* 参数非法（适配层校验失败） */
    XXX_ERR_INIT  = 3,   /* 未初始化 */
    XXX_ERR_RES   = 4,   /* 资源不可用（队列满/句柄空等） */
    XXX_ERR_TMO   = 5,   /* 超时 */
} xxx_status_t;
```

- 约定 `OK == 0`。`command.c` 当前 `OK == 1` 是历史包袱，迁移时归一到 `OK == 0`。
- **适配层负责错误码映射**：下层 `BSP_ZDT_ERR_PARAM` → 本层 `CHASSIS_ERR_PARAM`，不把下层枚举直接抛给上层。

## 8. 跨层调用时序示例（导航一次）

以"蓝牙下发路径导航目标"为例，展示各层与适配层如何协作：

```
[BLE 硬件] --RxEvent ISR--> 02_Service/it_dispatch (中断适配层:按句柄路由+长度检查)
   └─> ble_adaption(参数校验:帧头/长度) ─> ble.c(纯解析:取出 mode,x,y,yaw)
         └─> 02_Service/chassis_service: chassis_post_nav(mode,x,y,...)
               ├─ (适配:校验 mode 合法、坐标范围)
               ├─ mode=PATH → map_find_path()           [03_Middleware/map 纯 A*]
               │              (map_adaption 已注入场地/障碍/偏移)
               └─ xQueueOverwrite(命令队列)             [直接用 CMSIS-OS2]
[chassis_control_task 20ms 周期]
   └─> hwt101 get_latest(yaw)                            [04_Bsp 适配:deg]
   └─> chassis_adaption: deg→rad, pulse→编码器增量
   └─> mh_get_pos × 4 (临界区读 pos_pulse)               [04_Bsp handler]
   └─> mecanum_chassis_odom_update / _path_follow        [03_Middleware 纯运动学]
   └─> chassis_adaption set_motors: rps→int16 speed
         └─> mh_speed(speed,4,accel) → tx_queue          [04_Bsp handler 投递]
[zdt TX 线程]  取 tx_queue → 补够 3ms → 拼多机帧 → HAL UART DMA   [Core]
[zdt RX 线程]  取 rx_queue → zdt_motor_parse → 写 pos_pulse        [中断喂入]
```

每个 `─>` 跨层处都经过被调方适配层的参数校验，这就是"每层适配层做参数检查"的运行时形态。

## 10. 风险与注意事项

1. **IDE 工程重配（最大风险）**：按层重组会改动所有源文件路径，MDK-ARM `.uvprojx` 的 Groups、Include Paths 需重配；`.ioc` 重新生成只会覆盖 `Core/`、`Drivers/`，不动新分层目录（反而更安全）。**建议每阶段只移动少量文件并立即重新编译**。
2. **CubeMX 重生成**：`Core/`、`Drivers/`、`freertos.c` 仍由 CubeMX 管理，User Code 段以外会被覆盖。新分层目录独立于 CubeMX，互不干扰。`it_dispatch` 接管 ISR 回调后，注意别让 CubeMX 在别处又生成一份弱回调。
3. **命名长度 vs 模块前缀**：见 6.3，定稿时在"可读前缀"和"≤20 字符"间权衡，写 `.h` 时一次敲定避免反复改名。
4. **ISR 上下文约束**：`it_dispatch` 与各适配层的 ISR 路径**禁止阻塞调用**；与任务共享状态必须明确同步（现状用 `motorHandle` 互斥 + `FromISR` 队列，保留并审查）。
5. **`command.c` 的 `OK==1`**：归一到 `OK==0` 时，全量排查依赖旧布尔语义的调用点。
6. **纯逻辑层去全局**：现 `System_adaption.c` 里 `motor1~4`、`imu`、`mecanum_chassis` 是文件级全局，迁移后集中到组装根，对象指针经接口传递，避免散落全局。
7. **map 静态数组尺寸**：分辨率/场地尺寸是编译期常量（决定 `MAP_NODE_COUNT` 数组大小），放 `map_cfg.h` 由 `map.c` include，**不能**改成运行期注入。

- **command 精简**：原 60 条命令库精简为 `zdt_cmd` 实际所需（enable/speed/pos/home/report + 多机拼帧 + 返回帧解析）；完整库归档于 `00_Doc/ref/zdt_command_full/`，按需回捞。
- **上报周期维持 2ms**（不放宽，保里程计采样新鲜度）。
- **`zdt_decode` 处理全部返回帧**（位置上报 + 02/12/E2/EE/9F 应答），不止位置。
