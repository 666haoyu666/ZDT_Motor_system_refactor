# 里程计采集与容错升级说明

> 版本：草案 v2（2026-06-24）— 依据实际拓扑（共享 TX + 四路独立 RX）修订
> 范围：本次仅设计与说明，**不含代码改动**
> 适用：ZDT 麦轮底盘

## 0. 硬件拓扑（前提，决定方案）

- **TX 共享**：所有命令统一经 `huart3` TX 下发，物理上一条线扇出到四个电机的 RX 输入（广播）。
- **RX 独立**：电机 1~4 的回传分别走 `huart3 / huart4 / huart5 / huart6` 的 RX，各自 DMA + 空闲中断接收。
- **推论**：四个电机回复**互不冲突、可并行到达**；唯一串行资源是共享 TX
  （帧间受电机要求 ≥3ms 约束，因为每个电机都会收到 TX 上的每一帧）。

## 1. 升级概述

将里程计轮位采集由"各电机每 2ms 异步自动上报"改为"里程计线程按周期**广播同步读取**"：
每周期经共享 TX 发**一条广播读位置命令**，四个电机同一瞬间锁存编码器、在各自独立 RX 上并行回复；
四轮数据齐后统一积分更新里程计。配套引入超时重试、故障上报与采集 / 控制异步解耦。

## 2. 方案选择（结论）

| 方案 | 做法 | 结论 |
|------|------|------|
| **A（采用）** | 一条广播读（addr `0x00` / `0x36`）→ 四轮同刻锁存 → 并行回复 | **真正消除轮间相位偏差**，本次目标 |
| B（后备） | 保留 2ms 自动上报 + 每轮打时间戳补偿到同一时刻 | 广播读不可用时回退；无协议改动、风险最小 |
| C（排除） | 逐个寻址读四轮 | 共享 TX 上串行 ≥3ms，锁存摊到 ~9–12ms，**比现状更差，禁止** |

### ⚠️ 2.1 实施前置验证（方案 A 的成败关键）

**必须先确认：ZDT 电机是否对广播地址 `0x00` 的读位置命令回复。**

- 多数 RS485 协议广播命令"只执行不回复"。本拓扑 RX 独立、并行回复不会撞车，
  但**能否回复取决于电机固件**。
- 验证方式：查 ZDT / Emm 协议手册 `0x36` + 广播条目；或台架实测——发一条广播读，
  确认四路 RX 都收到回复。
- 验收：一条广播读，四个电机全部回复。
- **不通过 → 立即转方案 B，不得改用方案 C。**

## 3. 升级内容（基于方案 A）

### 3.1 采集方式
- 关闭电机端定时自动上报（原 2ms / `0x11`）。
- 里程计线程每周期经共享 TX 发**一条广播读位置帧**（`0x36`，addr `0x00`）。
- 四路独立 RX 并行收回复，四轮齐后统一脉冲差分 + 里程计积分。

### 3.2 容错与故障处置
- 限定时间内未集齐四轮 → 超时，**重新广播读**（重试一条帧即可）。
- **连续**超时达 10 次 → 经回调上报上层（回调只发信号、不做重活）。
- 上层独立轻量**错误监控线程**统一处置：**当前策略为简单停车**（竞赛先用，避免乱跑），
  恢复 / 重连策略后续迭代。

### 3.3 线程架构
- 里程计线程与底盘控制线程**异步解耦**：广播读的等待 / 超时阻塞隔离在里程计线程内，
  不拖累控制实时性。
  （注：方案 A 下并行回复，正常路径等待很短；异步主要为兜住超时 / 重试。）
- 里程计位姿成为跨线程共享数据，用**临界区**保护读写一致（锁置于底盘服务层）。

### 3.4 ⚠️ 上电就绪判定变更（重点）
- 原依据：收到电机自动上报即认为闭环在跑。
- 关闭自动上报后该信号消失，**改为：收到首次完整四轮广播读回复**才算就绪。
- **集成 / 产测的上电检测时序与脚本须同步调整**，否则会误判未就绪。

## 4. 升级收益（如实说明）

- **主要：四轮同一时刻采样 → 消除轮间相位偏差**，里程计积分更同步，旋转 / 加减速瞬态尤为明显。
- 次要：上报频度 2ms → 20ms，四路 RX 完成中断 / DMA 处理频度由约 2000/s 降到约 200/s，
  **降低 MCU 中断 / DMA 开销**。
- 采样与控制周期对齐，时序确定可预期。

> 客观说明：里程计用绝对累计脉冲做差分，**大致恒定的相位偏差本会自抵消**，
> 故方案 A 的提升量级在匀速段有限、在旋转 / 变速段明显。vx / vy 等速度量本产品当前未使用，不在目标内。

## 5. 兼容性

- 里程计算法、对外接口、坐标系定义不变，上层导航无需适配。
- 仅采集时序、容错路径、就绪握手变化（见 3.4）。

## 6. 实现要点（待落地，本次不写代码）

自底向上分层落地：

| 层 | 改动 |
|----|------|
| 协议层 `zdt_cmd` | 新增读位置帧构造（`0x36`），支持广播地址 `0x00` |
| 电机层 `zdt_motor` | 新增组级 `zdt_group_read_pos`（构广播读帧、经 `pf_send` 下发）；回复复用现有 `ZDT_RX_POS` 解析，无需新解析逻辑 |
| handler 层 | 新增 TX 请求类型 + 对外 `mh_request_pos_all`（一条广播读，仅入队，保持 OS 无关）|
| 服务层 | 里程计线程：清四位标志 → 广播读 → 带超时 `WaitAll` 四位 → 持锁读取并积分；超时重广播、连续 10 次回调；错误监控线程；位姿临界区 |
| 装配根 | 关闭自动上报（report 周期置 0）；就绪握手改为"首次四轮广播读回复" |

**同步原语**：四轮"等齐"用**事件标志（每轮一位，共四位）**、`WaitAll`；
不用互斥锁 / 二值信号量（无法区分是哪四个轮各自到位）。
RX 线程写完 `pos_pulse`（出锁）后置位；置位在线程上下文，普通置位接口即可。

**实现注意**：
1. 保留 handler 内护 `pos_pulse` 的临界区——int64 在 32 位 MCU 非原子；超时后迟到帧仍可能与读并发。
2. 位姿临界区**只护位姿拷贝（极短）**，**勿**把整个 `navigate` / `follow` 计算包进关中断的临界区；
   建议控制线程入口处持锁把位姿拷到局部，再用局部计算。
3. **RX 队列深度 ≥ 同时到达帧数(4) + 余量**：广播读后四帧几乎同时到，避免入队丢帧。
4. 广播读后四帧并行到达 → 四次 RX 完成中断 → 单 RX 线程逐帧解析（微秒级）；
   锁存已同步，解析顺序不影响采样值。
